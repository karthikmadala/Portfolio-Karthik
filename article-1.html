Introduction & Architecture Overview

Initial Coin Offerings (ICOs) remain a powerful fundraising mechanism in the blockchain ecosystem. Over the past year, building ICO platforms at App Stars Applications Pvt Ltd has given me deep insight into what it takes to ship a production-ready system that handles real money and real users.

In this guide, I'll walk you through the complete architecture of a Laravel-backed ICO platform with Web3.js integration. By the end, you'll understand how all the pieces fit together - from the smart contract on-chain to the admin panel in your browser.

This guide reflects the actual architecture we use in production at ico.zapme.co. The patterns here are battle-tested with thousands of real transactions.

High-Level Architecture

The system comprises four layers:

  • Frontend: HTML/CSS/JS with Web3.js for blockchain interactions
  • Backend: Laravel 11 for business logic, API, and admin panel
  • Blockchain: ERC-20 token contract on Ethereum/BSC
  • Payments: MoonPay, Transak, and Stripe for fiat-to-crypto onramping

Project Setup: Laravel + Web3.js

Start by creating a fresh Laravel 11 project and installing the necessary packages:

composer create-project laravel/laravel ico-platform
cd ico-platform
composer require web3p/web3.php
npm install web3 ethers @metamask/detect-provider

Configure your environment variables for blockchain connectivity:

INFURA_PROJECT_ID=your_infura_project_id
CHAIN_ID=1
TOKEN_CONTRACT_ADDRESS=0x...
ADMIN_WALLET_ADDRESS=0x...
MOONPAY_API_KEY=...
TRANSAK_API_KEY=...

Database Schema

The core tables you'll need are:

  • users - standard auth + wallet_address field
  • transactions - purchase records with tx_hash, amount, status
  • token_allocations - per-user token balances pre-distribution
  • ico_settings - configurable soft/hard caps, price, dates

Smart Contract Integration

The ICO contract exposes a buyTokens() payable function. On the Laravel side, we verify transaction hashes using the web3p library:

use Web3\Web3;
use Web3\Providers\HttpProvider;

class BlockchainService
{
    protected $web3;

    public function __construct()
    {
        $this->web3 = new Web3(
            new HttpProvider('https://mainnet.infura.io/v3/' . config('blockchain.infura_id'))
        );
    }

    public function verifyTransaction(string $txHash): array
    {
        $result = [];
        $this->web3->eth->getTransactionReceipt($txHash, function ($err, $receipt) use (&$result) {
            if ($err) throw new \Exception('TX verification failed: ' . $err->getMessage());
            $result = [
                'status'      => hexdec($receipt->status) === 1,
                'blockNumber' => hexdec($receipt->blockNumber),
                'from'        => $receipt->from,
                'to'          => $receipt->to,
            ];
        });
        return $result;
    }
}

MetaMask Connection Flow

The frontend wallet connection is the most user-facing piece. Here's the pattern that handles MetaMask across EVM networks gracefully:

import detectEthereumProvider from '@metamask/detect-provider';

async function connectWallet() {
  const provider = await detectEthereumProvider();
  if (!provider) {
    showError('MetaMask not found. Please install it from metamask.io');
    return;
  }

  // Request account access
  const accounts = await window.ethereum.request({
    method: 'eth_requestAccounts'
  });

  // Check we're on the right network
  const chainId = await window.ethereum.request({ method: 'eth_chainId' });
  if (chainId !== TARGET_CHAIN_ID) {
    await switchNetwork(TARGET_CHAIN_ID);
  }

  return accounts[0];
}

async function switchNetwork(targetChainId) {
  try {
    await window.ethereum.request({
      method: 'wallet_switchEthereumChain',
      params: [{ chainId: targetChainId }],
    });
  } catch (switchError) {
    // Chain not added yet - prompt user to add it
    if (switchError.code === 4902) {
      await window.ethereum.request({
        method: 'wallet_addEthereumChain',
        params: [CHAIN_CONFIG],
      });
    }
  }
}

Integrating Onramp Payment Gateways

We support three onramp providers to maximize conversion rates across different user geographies:

  • MoonPay - best coverage for Europe and US
  • Transak - strong coverage in India and Southeast Asia
  • Stripe - for credit/debit card payments where crypto isn't the preferred method

Each gateway receives a webhook callback when a payment completes. The Laravel controller validates the webhook signature and triggers the token allocation:

class OnrampWebhookController extends Controller
{
    public function moonpay(Request $request): JsonResponse
    {
        // Verify HMAC signature
        $signature = hash_hmac('sha256',
            $request->getContent(),
            config('services.moonpay.secret_key')
        );

        if (!hash_equals($request->header('Moonpay-Signature-V2'), $signature)) {
            return response()->json(['error' => 'Invalid signature'], 401);
        }

        $data = $request->json()->all();
        if ($data['status'] === 'completed') {
            TokenAllocationJob::dispatch($data['externalTransactionId']);
        }

        return response()->json(['received' => true]);
    }
}

Admin Panel Design

The admin panel is built with Laravel Livewire for real-time updates. Key features include:

  • Real-time transaction monitoring dashboard
  • Manual token allocation override
  • ICO phase management (soft cap, hard cap, price adjustment)
  • KYC verification workflow integration
  • Export to CSV for accounting

Production Deployment on AWS

Our production stack on AWS uses:

  • EC2 t3.medium - application server running Ubuntu 22.04 + Apache2
  • RDS MySQL 8.0 - managed database with automated backups
  • ElastiCache Redis - session storage and Laravel queue driver
  • SES - transactional email for purchase confirmations
  • CloudFront - CDN for static assets

Key deployment command after pulling latest code:

php artisan down
git pull origin main
composer install --no-dev --optimize-autoloader
php artisan migrate --force
php artisan config:cache
php artisan route:cache
php artisan view:cache
php artisan up

Conclusion

Building an ICO platform is a fascinating intersection of traditional web engineering and blockchain technology. The key lessons from our production experience:

  • Always verify transactions on-chain - never trust only the frontend
  • Design for idempotency - webhooks can fire multiple times
  • Keep smart contract interaction logic in a dedicated service class
  • Handle network switching gracefully - users may be on the wrong chain
  • Monitor your Infura/Alchemy rate limits in production

If you're building something similar, feel free to reach out - I'm always happy to discuss blockchain architecture patterns.
