Eliminating the N+1 Query Problem

The N+1 problem is the single most common performance killer in Laravel applications. It occurs when you load a collection and then lazily load a relationship for each item, firing one query per record instead of one joined query.

The Problem

// BAD - fires 1 + N queries
$users = User::all(); // 1 query: SELECT * FROM users
foreach ($users as $user) {
    echo $user->wallet->balance; // N queries: SELECT * FROM wallets WHERE user_id = ?
}

The Fix: Eager Loading

// GOOD - fires exactly 2 queries
$users = User::with('wallet')->get();
foreach ($users as $user) {
    echo $user->wallet->balance; // no extra queries
}

// BETTER - load only what you need
$users = User::with(['wallet:id,user_id,balance'])->get();

For conditional eager loading:

$users = User::when($request->has('with_transactions'), function ($query) {
    $query->with(['transactions' => function ($q) {
        $q->latest()->limit(10);
    }]);
})->paginate(20);

Strategic Indexing for Laravel Apps

Indexes are the most impactful optimization lever. The rule of thumb: index any column that appears in a WHERE, ORDER BY, or JOIN ON clause of a frequently-run query. 

// Migration: composite index for common query pattern
Schema::table('transactions', function (Blueprint $table) {
    // Index for: WHERE user_id = ? AND status = ? ORDER BY created_at DESC
    $table->index(['user_id', 'status', 'created_at'], 'idx_user_status_created');

    // Full-text index for search
    $table->fullText(['description', 'notes']);
});

Indexes you should always have in a blockchain application:

  • transactions.tx_hash - unique, used in every TX lookup
  • transactions.wallet_address - for wallet history queries
  • transactions.status + created_at - for pending TX polling
  • users.email - unique, used in every auth query

Reducing Memory with select()

Fetching SELECT * is wasteful when you only need two columns. The select() clause saves both memory and network transfer: 

// Wasteful - transfers all 30 columns per row
$transactions = Transaction::where('status', 'pending')->get();

// Efficient - transfers only what the view needs
$transactions = Transaction::select(['id', 'user_id', 'amount', 'tx_hash', 'created_at'])
    ->where('status', 'pending')
    ->latest()
    ->get();

// Even better for huge tables - use cursor() for memory efficiency
Transaction::where('status', 'pending')
    ->select(['id', 'tx_hash'])
    ->cursor()
    ->each(function ($tx) {
        ProcessTransaction::dispatch($tx);
    });

Chunking Large Datasets

When processing thousands of rows, never call get() - it loads everything into memory at once. Use chunk() or chunkById(): 

// BAD - loads 50,000 rows into PHP memory simultaneously
$transactions = Transaction::where('status', 'unprocessed')->get();

// GOOD - processes 500 at a time
Transaction::where('status', 'unprocessed')
    ->chunkById(500, function ($transactions) {
        foreach ($transactions as $tx) {
            // process each transaction
        }
    });

// BEST for ordered data - chunkById is safer than chunk
// chunk() can miss/duplicate rows if records are updated during iteration

Query Caching with Redis

Some queries are expensive and their results don't change frequently. Cache them: 

// Cache ICO settings - they change rarely
$icoSettings = Cache::remember('ico:settings', now()->addHours(1), function () {
    return IcoSetting::first();
});

// Cache per-user token balance
$balance = Cache::remember("user:{$userId}:token_balance", 300, function () use ($userId) {
    return TokenAllocation::where('user_id', $userId)->sum('tokens');
});

// Invalidate cache after update
public function updateAllocation(User $user, float $tokens): void
{
    DB::transaction(function () use ($user, $tokens) {
        TokenAllocation::updateOrCreate(
            ['user_id' => $user->id],
            ['tokens' => DB::raw("tokens + {$tokens}")]
        );
        Cache::forget("user:{$user->id}:token_balance");
    });
}

Reading EXPLAIN Output

When a query is still slow despite your efforts, EXPLAIN tells you exactly what MySQL is doing: 

EXPLAIN SELECT t.*, u.email
FROM transactions t
JOIN users u ON u.id = t.user_id
WHERE t.status = 'completed'
  AND t.created_at > '2026-01-01'
ORDER BY t.created_at DESC
LIMIT 50;

Key things to look for in the output:

  • type = ALL - full table scan, needs an index
  • type = ref or eq_ref - good, using an index
  • Extra = Using filesort - sorting without index, consider adding one
  • rows - estimated rows examined; lower is better

Profiling Tools

The tools I use daily for query performance monitoring:

  • Laravel Debugbar - shows all queries per request in development
  • Laravel Telescope - production query logging with slow query detection
  • MySQL slow query log - set long_query_time = 1 to catch slow queries
  • EXPLAIN ANALYZE (MySQL 8.0+) - actual vs estimated row counts 
// In AppServiceProvider - log slow queries in local env
if (app()->isLocal()) {
    DB::listen(function ($query) {
        if ($query->time > 500) { // ms
            Log::warning('Slow query detected', [
                'sql'  => $query->sql,
                'time' => $query->time . 'ms',
                'bindings' => $query->bindings,
            ]);
        }
    });
}
