EVM Fundamentals for DApp Developers

The Ethereum Virtual Machine (EVM) is a standardized execution environment that has been adopted by dozens of blockchains. This standardization means that a wallet built for Ethereum works on BSC, Polygon, Avalanche, and hundreds of other EVM-compatible chains - with just a network switch.

The key identifiers you'll work with constantly are:

  • Chain ID - unique identifier for each network (1 = Ethereum mainnet, 56 = BSC, 137 = Polygon)
  • RPC URL - the JSON-RPC endpoint for your node provider
  • Address format - always 0x-prefixed 40-character hex strings (checksummed)

The MetaMask Provider API

MetaMask injects a window.ethereum object into every webpage. This object implements the EIP-1193 provider specification. Here's the complete connection boilerplate I use in production:

class WalletConnector {
  constructor() {
    this.provider = null;
    this.account = null;
    this.chainId = null;
  }

  async init() {
    if (!window.ethereum) {
      throw new Error('NO_WALLET');
    }

    this.provider = window.ethereum;

    // Listen for account changes
    this.provider.on('accountsChanged', (accounts) => {
      this.account = accounts[0] || null;
      this.onAccountChange?.(this.account);
    });

    // Listen for chain changes
    this.provider.on('chainChanged', (chainId) => {
      this.chainId = chainId;
      // Reload recommended by MetaMask docs
      window.location.reload();
    });
  }

  async connect() {
    const accounts = await this.provider.request({
      method: 'eth_requestAccounts'
    });
    this.account = accounts[0];
    this.chainId = await this.provider.request({ method: 'eth_chainId' });
    return this.account;
  }

  async signMessage(message) {
    return await this.provider.request({
      method: 'personal_sign',
      params: [message, this.account],
    });
  }
}

Multi-Chain Support: Ethereum, BSC, Polygon

Supporting multiple EVM chains requires maintaining a config map and gracefully prompting users to switch:

const CHAIN_CONFIG = {
  '0x1': {
    name: 'Ethereum Mainnet',
    rpcUrls: ['https://mainnet.infura.io/v3/YOUR_KEY'],
    nativeCurrency: { name: 'Ether', symbol: 'ETH', decimals: 18 },
    blockExplorerUrls: ['https://etherscan.io'],
  },
  '0x38': {
    name: 'BNB Smart Chain',
    rpcUrls: ['https://bsc-dataseed.binance.org/'],
    nativeCurrency: { name: 'BNB', symbol: 'BNB', decimals: 18 },
    blockExplorerUrls: ['https://bscscan.com'],
  },
  '0x89': {
    name: 'Polygon Mainnet',
    rpcUrls: ['https://polygon-rpc.com'],
    nativeCurrency: { name: 'MATIC', symbol: 'MATIC', decimals: 18 },
    blockExplorerUrls: ['https://polygonscan.com'],
  },
};

async function ensureNetwork(targetChainId) {
  const currentChainId = await window.ethereum.request({ method: 'eth_chainId' });
  if (currentChainId === targetChainId) return;

  try {
    await window.ethereum.request({
      method: 'wallet_switchEthereumChain',
      params: [{ chainId: targetChainId }],
    });
  } catch (error) {
    if (error.code === 4902) {
      // Network not in MetaMask - add it
      await window.ethereum.request({
        method: 'wallet_addEthereumChain',
        params: [{ chainId: targetChainId, ...CHAIN_CONFIG[targetChainId] }],
      });
    } else {
      throw error;
    }
  }
}

Non-EVM Wallets: The Sui Approach

Sui uses a completely different wallet model - the Move VM, object-centric storage, and a different address format. The Sui Wallet browser extension exposes window.suiWallet (or the newer @mysten/wallet-standard).

// Sui Wallet connection
async function connectSuiWallet() {
  // Check for Sui Wallet extension
  if (!window.suiWallet) {
    // Fallback to wallet-standard
    const { getWallets } = await import('@mysten/wallet-standard');
    const wallets = getWallets().get();
    const suiWallet = wallets.find(w => w.name.includes('Sui'));
    if (!suiWallet) throw new Error('Sui Wallet not found');
  }

  const response = await window.suiWallet.requestPermissions({
    permissions: ['viewAccount', 'suggestTransactions'],
  });

  if (response.result) {
    const accounts = await window.suiWallet.getAccounts();
    return accounts[0].address; // 0x-prefixed 32-byte hex
  }
}

// Signing a Sui transaction
async function executeSuiTransaction(txBytes) {
  const result = await window.suiWallet.signAndExecuteTransactionBlock({
    transactionBlock: txBytes,
    options: {
      showEffects: true,
      showObjectChanges: true,
    },
  });
  return result.digest; // transaction hash
}

Key Differences: EVM vs Sui

The most important architectural differences to account for:

  • Address format: EVM uses 20-byte hex; Sui uses 32-byte hex
  • Transaction model: EVM is account-based; Sui is object-centric (UTXO-like)
  • Gas: EVM uses ETH/native token; Sui uses SUI for gas
  • Finality: Sui achieves finality in ~500ms vs Ethereum's ~12s

Production Error Handling Patterns

Wallet interactions are full of edge cases. Here's the error taxonomy I've learned to handle:

const WALLET_ERRORS = {
  4001: 'User rejected the request',
  4100: 'Unauthorized - connect wallet first',
  4200: 'Unsupported method',
  4900: 'Disconnected from chain',
  4901: 'Chain not connected',
  4902: 'Chain not added to wallet',
  '-32002': 'Request already pending - check MetaMask popup',
  '-32603': 'Internal JSON-RPC error',
};

function handleWalletError(error) {
  const message = WALLET_ERRORS[error.code]
    || WALLET_ERRORS[String(error.code)]
    || error.message
    || 'Unknown wallet error';

  // User rejections are not bugs - don't log them as errors
  if (error.code === 4001) {
    showInfo(message);
    return;
  }
  showError(message);
  console.error('[Wallet Error]', error);
}

Building a Unified Wallet Abstraction Layer

When your DApp needs to support both EVM and Sui, a unified abstraction layer is essential. The interface should expose the same methods regardless of the underlying wallet:

class WalletManager {
  constructor() {
    this.adapters = {
      evm: new EVMWalletAdapter(),
      sui: new SuiWalletAdapter(),
    };
    this.active = null;
  }

  async connect(type = 'evm') {
    const adapter = this.adapters[type];
    const address = await adapter.connect();
    this.active = { type, adapter, address };
    return address;
  }

  async getBalance() {
    return this.active.adapter.getBalance(this.active.address);
  }

  async sendTransaction(params) {
    return this.active.adapter.sendTransaction(params);
  }
}

This pattern keeps your UI components ignorant of wallet specifics - they just call walletManager.sendTransaction() and the adapter handles the rest.
